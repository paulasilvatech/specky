# Auditoria Técnica e de Produto — Specky (`specky-sdd`)

> **Data:** 2026-07-01 · **Versão auditada:** `3.4.0-rc.15` (branch `main`/`develop`, commit `764f42b`)
> **Escopo:** código-fonte (`src/`, 88 arquivos TS, ~19.000 LOC), testes, CI/CD, segurança, assets (`.apm/`), documentação, claims públicos, execução real do pipeline via MCP, e posicionamento no ecossistema SDD.
> **Método:** análise estática + execução dinâmica (build, suíte de testes, servidor MCP real via JSON-RPC, instalação em projeto limpo) + verificação de cada claim público contra o código + pesquisa do ecossistema SDD 2025–2026.

---

## 1. Sumário Executivo

O Specky é um projeto **ambicioso e com fundação real**: 58 tools MCP registradas e funcionais, validação Zod `.strict()` de verdade em todas as entradas, máquina de estados persistida, camada de enforcement com RBAC/auditoria via decorator, CLI multi-IDE que funciona, e uma distribuição npm com provenance, cosign e SBOM. Isso é mais engenharia do que a maioria dos toolkits SDD possui hoje.

Porém, a auditoria encontrou um padrão consistente: **a distância entre o que o produto promete ("determinismo, enforcement, gates confiáveis") e o que o motor entrega**. Os problemas mais graves não são cosméticos — estão exatamente no coração da proposta de valor:

| # | Achado crítico | Severidade |
|---|---|---|
| 1 | `sdd_auto_pipeline` **fabrica** um `ANALYSIS.md` com `gate_decision: "APPROVE"`, cobertura 100% e zero órfãos — hardcoded, sem executar análise real (`src/tools/transcript.ts:630-635`) | 🔴 Crítica |
| 2 | **Split de identidade de feature**: `sdd_init` registra `.specs/001-<project_name>` no estado, mas `sdd_write_spec` grava em `.specs/001-<slug(feature_name)>`. O pipeline **trava no caminho feliz** ("missing SPECIFICATION.md") e o estado marca `specify: completed` mesmo assim | 🔴 Crítica |
| 3 | `specky install` **pré-autoriza silenciosamente** `Bash(bash:*)`, `Bash(sh:*)`, `Bash(rm:*)`, `Write`, `WebFetch`, `WebSearch` no `.claude/settings.json` do projeto do usuário — desliga o modelo de permissões do Claude Code | 🔴 Crítica |
| 4 | **CI do `main` está vermelho**: teste exige `Dockerfile` que nunca foi commitado em nenhum branch (`tests/unit/docs-enterprise.test.ts:60`); o job Docker do `publish.yml` falharia pelo mesmo motivo. _(Resolvido neste PR: `Dockerfile` + `.dockerignore` reais adicionados; suíte 113/113 verde.)_ | 🔴 Crítica |
| 5 | **Claims inflados**: README anuncia "507 tests" — existem **113** (1 falhando); cobertura anunciada (56%) é medida só sobre arquivos importados pelos testes (~1.330 de ~19.000 linhas instrumentadas); seção "All 58 Tools" lista 57 | 🟠 Alta |
| 6 | npm `latest` aponta para um **release candidate** (rc.15), violando a regra do próprio `scripts/release.mjs`; não existe nenhuma git tag no repositório | 🟠 Alta |
| 7 | Estado sem lock e escrita não-atômica do par state+sig; audit log com hash chain sujeito a fork em concorrência | 🟠 Alta |
| 8 | Zero **dogfooding**: o repositório do Specky não tem `.specs/` — a ferramenta de SDD não é usada para desenvolver a si mesma; não há `examples/` | 🟠 Alta (produto) |

A seção 11 traz o roadmap priorizado.

## 2. Metodologia da auditoria

1. **Execução real**: `npm ci`, `npm run build`, `vitest run --coverage`; servidor MCP iniciado e exercitado via JSON-RPC puro (initialize → tools/list → tools/call) num workspace limpo; `specky install --ide=claude` em projeto vazio; inspeção dos artefatos gerados.
2. **Análise estática paralela**: 4 auditorias independentes (arquitetura, claims vs realidade, testes/CI, segurança) com evidência `arquivo:linha` para cada achado.
3. **Pesquisa de ecossistema**: levantamento do estado da arte SDD 2025–2026 (GitHub Spec-Kit, AWS Kiro, OpenSpec, Tessl, BMAD e afins) para calibrar o que "ser referência" exige.
4. **Verificação cruzada**: todo achado de agente relevante foi re-confirmado manualmente no código antes de entrar neste relatório.

## 3. O que está genuinamente bom (manter e defender)

- **58 tools reais**: `tools/list` retorna exatamente 58; guard de regressão `tests/unit/tool-count.test.ts` impede novo drift de contagem.
- **Camada de schemas exemplar**: Zod `.strict()` em tudo, regex, min/max, `.describe()`; zero `z.any()`/`.passthrough()` (`src/schemas/*`). Rejeição de chaves desconhecidas verificada em runtime.
- **Composition root limpo** (`src/index.ts:102-167`): serviços construídos uma vez e injetados; `FileManager` como chokepoint de I/O; engines puras e testáveis.
- **Enforcement transversal elegante**: `registerTool` decorado antes do registro (`src/tools/tool-enforcement.ts:208-220`) aplicando RBAC → validação de fase → auditoria hash-chained em todas as tools.
- **Type safety acima da média**: zero `: any` real no `src/` (gate de CI via grep), ~8 non-null assertions no total.
- **Testes de integração que falam MCP de verdade**: 4 suítes spawnam `dist/index.js` e conversam JSON-RPC real — padrão raro e correto.
- **Supply chain do release**: `npm publish --provenance`, cosign no container, SBOM CycloneDX (`.github/workflows/publish.yml`), actions pinadas por SHA, Dependabot npm+actions, OpenSSF Scorecard.
- **Instalador multi-IDE funcional**: 62 arquivos instalados de forma idempotente com manifest de integridade (`.specky/install.lock`) e `specky doctor` para validação.
- **Determinismo operacional**: seam `SDD_FIXED_NOW` para timestamps estáveis; escrita atômica por arquivo (temp+rename) no `FileManager`.
- **Documentação extensa e bem organizada** (README, GETTING-STARTED, 10 docs enterprise) — o problema é acurácia, não esforço.

## 4. Achados críticos detalhados (P0)

### 4.1 Gate fabricado no `sdd_auto_pipeline` — quebra o contrato central do produto

`src/tools/transcript.ts:626-648`: ao gerar o pacote de spec a partir de transcript, o tool escreve `ANALYSIS.md` com:

```ts
gate_decision: "APPROVE",
coverage_percent: "100",
traceability_matrix: "| All docs | Present | — | — | ✅ |",
design_coverage: "100%",
task_coverage: "100%",
orphan_count: "0",
```

— tudo constante, sem executar `sdd_run_analysis` nem a matemática real de gate (`src/tools/pipeline.ts:809-837`). O documento que o usuário lê como "quality gate aprovado com 100% de cobertura" é fabricado. Além de bug de produto, é exatamente o tipo de "AI theater" que o Specky diz combater. **Correção:** rotear `auto_pipeline`/`batch_transcripts` pelos mesmos serviços de análise do pipeline normal e gravar o resultado verdadeiro (inclusive `CHANGES_NEEDED`/`BLOCK`).

### 4.2 Split de identidade de feature — o caminho feliz trava

Reproduzido em execução real (workspace limpo, sequência `sdd_init` → `sdd_discover` → `sdd_advance_phase` → `sdd_write_spec` → `sdd_advance_phase`):

- `sdd_init { project_name: "user-auth" }` cria e registra no estado `.specs/001-user-auth` (`state.features[0]`).
- `sdd_write_spec { feature_name: "User Authentication", ... }` deriva o diretório **do slug do feature_name** (`src/tools/pipeline.ts:275`) e grava tudo em `.specs/001-user-authentication/` — diretório que o estado não conhece.
- O `sdd_advance_phase` seguinte falha: `Cannot advance: missing required files: SPECIFICATION.md` — porque `checkRequiredArtifacts` olha `state.features[0]` (`src/services/state-machine.ts:157`).
- Pior: o estado fica **inconsistente** — `phases.specify.status: "completed"` foi gravado por `recordPhaseComplete` (caminho de mutação sem validação, `state-machine.ts:258-266`) enquanto o caminho validado (`advancePhase`) rejeita a transição.

Três causas-raiz a corrigir: (a) identidade da feature deve vir **do estado**, nunca ser re-derivada de nomes; (b) slugificação duplicada e divergente (`pipeline.ts:275` não remove caracteres especiais; `turnkey.ts:89` remove); (c) dois caminhos paralelos de mutação de estado (com e sem validação).

### 4.3 Permissões pré-autorizadas amplas no projeto do usuário

`specky install --ide=claude` grava em `.claude/settings.json` do projeto (verificado em instalação real):

```json
"allow": ["Read","Glob","Grep","Edit","Write","MultiEdit","WebFetch","WebSearch","Task",
 "Bash(bash:*)","Bash(sh:*)","Bash(git:*)","Bash(npm:*)","Bash(rm:*)","Bash(chmod:*)",
 "Bash(pip:*)","Bash(python:*)", …, "mcp__specky__*"]
```

`Bash(bash:*)` + `Bash(sh:*)` + `Bash(rm:*)` + `Write` + `WebFetch` = o agente executa qualquer comando, apaga qualquer coisa e acessa a rede **sem nunca pedir confirmação**. Para um produto vendido como "governança enterprise" e "zero outbound network", instalar isso silenciosamente é a contradição mais séria do projeto — e mina a confiança de qualquer avaliação de segurança corporativa. **Correção:** permitir por padrão apenas `mcp__specky__*` (+ leitura), tornar o resto opt-in explícito (`--permissions=full` com aviso), e documentar o que é gravado.

### 4.4 CI vermelho no `main` + release quebrado

- `tests/unit/docs-enterprise.test.ts:60-63` exige `Dockerfile` e `.dockerignore` na raiz; o arquivo **nunca existiu** em nenhum branch. A suíte falha hoje em `main` e `develop` (verificado: `1 failed | 112 passed`).
- **Causa-raiz identificada:** o próprio `.gitignore` (`:47,49`) listava `Dockerfile` e `.dockerignore` como "build config (internal)" — provavelmente porque `sdd_generate_dockerfile` os gera em projetos de usuário. Um commit posterior adicionou o teste + o job `docker-publish` esperando um `Dockerfile` **rastreado**, e o `.gitignore` vencia silenciosamente essa contradição, tornando impossível commitar o artefato. (Corrigido neste PR: `Dockerfile`/`.dockerignore` destravados; `docker-compose.yml` segue ignorado por ser gerado por projeto.)
- O CHANGELOG do rc.15 afirma "Added Dockerfile and .dockerignore for the GHCR publish workflow" — o item descreve algo que não está no repositório.
- Consequências em cascata: `publish.yml` (job Docker/GHCR + cosign + SBOM) falharia; a seção Development do README (`docker build -t specky-sdd:dev .`) não funciona.
- Agrava: npm `latest` = `3.4.0-rc.15` (dist-tags verificados no registry), embora `scripts/release.mjs:192` bloqueie prerelease→latest e o PUBLISH.md prescreva `--tag next`. E `git tag --list` está vazio — nenhuma release é rastreável a um commit.

> **Status neste PR:** o `Dockerfile` prometido (multi-stage, non-root, `EXPOSE 3200`, `HEALTHCHECK` no `/health`, `CMD … serve --http`) e o `.dockerignore` foram adicionados, satisfazendo `docs-enterprise.test.ts` e o job `docker-publish` do `publish.yml`. Validação sem daemon (registry docker.io bloqueado no sandbox da auditoria): o build stage compila usando apenas os diretórios que o Dockerfile copia; o runtime stage instala só deps de produção, sobe o servidor HTTP na 3200 e o `/health` responde `{"status":"ok"}`. Os itens de release restantes (dist-tag `latest`→estável, tags git, `prepublishOnly`) seguem em aberto.

## 5. Arquitetura e qualidade de código

### 5.1 Visão geral

Fluxo de dependência limpo e unidirecional:

```
index.ts (composition root, DI)
  └─> tools/*.ts (58 handlers, registrados com enforcement wrapper)
        └─> services/*.ts (~29 serviços; FileManager centraliza I/O)
              ├─> schemas/*.ts (Zod .strict())
              ├─> utils/*, constants.ts, types.ts
              └─> templates/ (22 skeletons via TemplateEngine)
```

CLI separado (`src/cli/index.ts`) com lazy `import()` por subcomando. Engines puras sem dependências de construtor (`ComplianceEngine`, `ModelRoutingEngine`, `CognitiveDebtEngine`, etc.) — trivialmente testáveis (e ainda assim não testadas, ver §6).

### 5.2 Problemas principais

1. **Duplicação como problema dominante**
   - `formatError` copiado em **16** arquivos de tools; `truncate` em **17**; envelope MCP `{ content: [{ type: "text", text: JSON.stringify(...) }] }` inline **127×** em 18 arquivos. Um único módulo `toTextResult()/toErrorResult()` elimina centenas de linhas.
   - Bloco de validação de fase copiado 6× dentro de `pipeline.ts` (151-164, 260-273, 377-390, 497-510, 630-643, 727-740) — **código morto**, pois o wrapper de enforcement já valida centralmente (`tool-enforcement.ts:182-193`).
   - `sdd_auto_pipeline` (`transcript.ts:534-644`) reimplementa o pipeline inteiro inline em vez de reusar os serviços — origem do gate fabricado (§4.1).

2. **God-files**: `transcript.ts` (1.006 LOC), `pipeline.ts` (960), `transcript-parser.ts` (911), `diagram-generator.ts` (694). `state-machine.ts` (512) mistura persistência + transições + policy map de 60 tools + heurística de completude de design.

3. **Estado e auditoria sem atomicidade/lock**
   - `saveState` escreve `.sdd-state.json` e depois `.sig` separadamente (`state-machine.ts:84-90`) — crash entre os dois gera falso "tamper detected".
   - Nenhum mutex: read-modify-write concorrente (modo `--http` ou chamadas paralelas) perde atualizações silenciosamente; o hash chain do audit log (`audit-logger.ts:187-200`) forka sob concorrência.
   - `loadState` faz **escrita** de migração em disco (side effect num read, `state-machine.ts:60-73`).
   - Uma chamada de `sdd_write_spec` dispara ~5-6 `loadState` + 4 `saveState` — janela de corrida ampliada e I/O redundante.

4. **Conteúdo hardcoded vs templates**: só ~10 dos 22 templates são usados na geração; `doc-generator.ts` e `iac-generator.ts` embutem runbook/Dockerfile/Terraform em strings TS (drift garantido). Template `onboarding` declarado em `TEMPLATE_NAMES` (`constants.ts:347`) **sem arquivo** → `sdd_get_template("onboarding")` lança erro; `templates/test-stub.md` existe e não é referenciado.

5. **Código morto e dependência fantasma**: bloco `TOOL_*` (~30 consts, `constants.ts:284-326`) não usado; `buildPhaseError` exportado e nunca chamado; dep `yaml` declarada e nunca importada (parser manual em `config.ts:109-165` que não trata listas aninhadas/quotes/comentários); `discovery_answers` é `required` no schema e o handler nunca lê.

6. **tsconfig aquém do possível**: `strict: true` mas sem `noUncheckedIndexedAccess`, `noUnusedLocals/Parameters`, `exactOptionalPropertyTypes` — que exporiam `undefined`s latentes (`pipeline.ts:219 questions[2]`, acessos `match[1]`, `PHASE_ORDER[i+1]`) e o código morto acima.

7. **Validador EARS raso para o peso que o marketing coloca nele** (`src/services/ears-validator.ts`, 193 LOC):
   - Termos vagos só são checados **quando o padrão casa** (early return em :79-88) — "The system should be fast" não recebe flag de vagueza.
   - `extractAction` não trata "shall" (:180) → sugestão duplicada: `"The system shall The system shall be fast."` (reproduzido via `sdd_validate_ears`).
   - Padrão **complex é inalcançável**: a regex de state_driven (prioridade 2) captura "While …, when …, the system shall …" antes da complex (prioridade 5).
   - Lista de vagos com substring match (7 termos; "fast" casaria "breakfast"); sem checagem de múltiplos requisitos numa frase, voz passiva, escape words ("etc", "as appropriate"), unidades mensuráveis.
   - Zero testes unitários para este serviço — o coração declarado do produto.

8. **Heurísticas de qualidade frouxas**: `validateDesignCompleteness` (`state-machine.ts:454-511`) pontua por substring ("data", "api", "error", "context") — quase qualquer texto passa; gera falsa sensação de gate.

9. **Saída gerada com acabamento fraco**: `SPECIFICATION.md` gerado contém seções "2. Functional Requirements" e "3. Non-Functional Requirements" **vazias** (todos os requisitos caem em "Core Requirements") e `---` órfãos — verificado em execução real. Para um produto cujo artefato-vitrine é a spec, o template precisa ser impecável.

10. **Model routing hardcoded**: `sdd_advance_phase` devolve `fallback_chain: ["claude-opus-4-7", …, "gpt-4.5", "gpt-5"]` — IDs de modelo desatualizados/inexistentes hardcoded no código (`config.yml`/engine), envelhecendo mal. O CHANGELOG removeu modelos hardcoded dos agents; o motor ainda os emite.

## 6. Testes e CI/CD

### 6.1 Estado real

- **113 casos em 17 arquivos** (86 unit / 27 integration). Onde existem, são bons: `rbac-engine` (23 casos), `audit-logger` (14), `rate-limiter` (11, fake timers corretos), 4 suítes falando JSON-RPC real com o binário compilado.
- **Buracos**: 19/29 services sem teste (incl. `ears-validator`, `transcript-parser` 911 LOC, `diagram-generator`, `iac-generator`, `compliance-engine`, `cross-analyzer`); 17/20 arquivos de tools sem teste; CLI com só 2 casos (`doctor`); todos os 15 schemas sem teste dedicado; **5 de 58 tools** exercitadas via protocolo MCP.
- **Cobertura mal medida**: sem `coverage.include: ["src/**"]`, o v8 só instrumenta arquivos importados → thresholds (50/40/60/50) avaliados sobre ~7% das linhas do projeto. O número público de cobertura não representa o codebase.
- `docs-enterprise.test.ts` são testes de string sobre docs (pin de contagens) — úteis como guard, mas inflam contagem sem exercitar lógica; e um deles está falhando (§4.4).

### 6.2 CI

- `ci.yml` (ubuntu + node 22 apenas): audit high, build, guard de frontmatter, testes+cobertura, grep anti-`: any`, handshake MCP. CodeQL **roda em PR** via *default setup* do GitHub (checks `Analyze (javascript-typescript)`/`Analyze (actions)` — não é um workflow file no repo, é configuração de repositório; confirmado nos checks do PR desta auditoria). Faltam, ainda assim: **ESLint/Prettier** (o grep anti-`: any` é o único "lint"), **`tsc --noEmit` dedicado**, e **matriz OS/Node no gate principal** (a matriz existe só no `install-smoke.yml`, path-filtered — mudanças em `src/services/**` não a disparam). O Scorecard roda só em schedule.
- `hooks-compat.yml` é um destaque positivo (portabilidade bash ubuntu+macos, ban de bashisms).
- Riscos de flake: integração MCP com `spawnSync` timeout 10s (o suite de hooks já precisou de 20s no Windows).
- Sem `CODEOWNERS`; Dependabot não cobre ecossistema `docker`.

### 6.3 Release

- `release.mjs` é um bom pre-flight, mas **não está wired** em `prepublishOnly` — nada força sua execução antes de `npm publish`.

## 8. Produto e metodologia SDD — o que falta para ser referência

_(Ver §10 para o quadro competitivo que fundamenta esta seção.)_

1. **Dogfooding zero é o gap mais visível.** O repositório do Specky não contém `.specs/` — nenhuma feature do Specky foi especificada com o Specky. Para uma ferramenta cujo pitch é "spec antes de código", isso é a primeira coisa que um avaliador cético procura. **Ação:** especificar as próximas features do próprio Specky com o pipeline completo e commitar os pacotes de spec; linkar do README.
2. **Sem exemplos públicos.** Não há `examples/` com um pacote `.specs/001-*` completo (spec → design → tasks → análise → verificação) que alguém possa ler em 10 minutos para "sentir" o produto. **Ação:** 2-3 exemplos canônicos (API REST greenfield; feature brownfield; modernização) commitados e renderizados.
3. **Nenhuma evidência de resultado.** "Teams spend 40% of their time on rework" (README:130) não tem fonte; não há benchmark, estudo de caso ou eval mostrando que specs+gates melhoram o resultado do agente (ex.: taxa de aceite de PR, retrabalho, drift). O docs/EVIDENCE.md é sobre integridade do repo, não sobre eficácia. **Ação:** eval reproduzível (mesmo pequeno: N tarefas com/sem Specky, mesmo modelo) publicado no repo — isso, mais que qualquer feature, cria autoridade técnica.
4. **A qualidade do artefato-vitrine precisa ser impecável** (§5.2 item 9): seções vazias e sugestões EARS bugadas minam a credibilidade do artefato central.
5. **Multi-feature é promessa não entregue**: README promete features simultâneas independentes; o estado é global (um `current_phase` por workspace; validação sempre em `features[0]`; `.sdd-state.json` na raiz de `.specs/`, não por feature como o próprio README desenha em :255-275). **Ação:** estado por feature (`.specs/NNN-*/.sdd-state.json`) com índice global leve — é também o que destrava paralelismo de equipes.
6. **Gates LGTM são soft fora do Analyze**: só Analyze→Implement tem gate rígido no motor (`state-machine.ts:204-221`); os gates de Specify/Design/Tasks dependem do agente obedecer o prompt/hook advisory. O README vende "actual gates that block advancement". **Ação:** ou implementar gate de aprovação persistida (ex.: `sdd_approve_gate` gravando decisão humana no estado, exigida pelo `advance_phase`), ou ajustar a linguagem do README.
7. **Lineage transparente do Spec-Kit**: o README aponta para `paulasilvatech/spec-kit` como "a metodologia"; o projeto original e amplamente conhecido é o `github/spec-kit`. Para credibilidade (e SEO honesto), declarar a relação com o projeto upstream explicitamente.
8. **Rastreabilidade de ponta a ponta ainda é parcial**: existe REQ→test (traceability mapper), mas não REQ→código (annotations/commits) nem verificação contínua pós-merge (drift check em CI). **Ação:** GitHub Action `specky-verify` que roda `sdd_check_sync`/`sdd_verify_tests` em PR — transformaria o Specky de tool de sessão em guardião contínuo (nenhum concorrente faz isso bem hoje).
9. **i18n é oportunidade real** (roadmap v3.5 já lista PT-BR/ES/…): nenhuma ferramenta SDD relevante fala português; para a comunidade BR isso é diferencial de adoção imediato.

### Leituras que definem o alvo

- **O mercado premia o leve, não o pesado.** OpenSpec (58k★) cresceu justamente rejeitando gates rígidos ("fluid not rigid"); a crítica mais repetida ao spec-kit é "ilusão de trabalho" e "um mar de markdown" (issue #75 do próprio spec-kit; Böckeler/Thoughtworks; Scott Logic: "reinvented waterfall"). Um estudo citado mediu 33 min + 2.577 linhas de markdown para produzir 689 LOC vs 8 min de prompting iterativo sem ganho de qualidade — a cerimônia escala pior em tarefas pequenas. **Implicação direta para o Specky:** 10 fases obrigatórias para todo tipo de mudança é remar contra a maré; precisa de um fast-path de 2 artefatos para mudanças pequenas e reservar a máquina completa para trabalho grande/regulado.
- **Brownfield é o mercado real.** O `/speckit.converge` (novo) e a tese inteira do OpenSpec (deltas) mostram que "mudar sistema existente" > greenfield. O Specky tem `sdd_scan_codebase`/`sdd_check_sync`, mas o fluxo é greenfield-primeiro.
- **Verificação de circuito fechado é o único antídoto para "ilusão de trabalho".** O diferencial da Kiro é property-based testing de código-vs-spec. O Specky tem `sdd_generate_pbt` e traceability REQ→teste, mas não fecha o loop em CI (drift check contínuo). Aqui há espaço para liderar.
- **Aprovação humana precisa de UX de revisão.** O spec-workflow-mcp (4.3k★, primo direto do Specky) provou que gates LGTM querem um dashboard/diff revisável, não confirmação no terminal.
- **Ninguém publica evidência de resultado.** Nenhuma ferramenta SDD publica benchmarks de retrabalho/defeito. Já existe infra acadêmica (SANER 2026, CodeSpecBench). **O primeiro a publicar auto-benchmark crível define o placar da categoria.**

### Padrões que importam em 2026

- **EARS** foi mainstreamado para specs de IA **pela Kiro** (não pelo spec-kit). Fato relevante: o **upstream `github/spec-kit` não usa EARS, não enforça fases (são prompts) e não tem "25+ agentes"** — o número real são 25/30+ *assistentes suportados*.
- **MCP** agora é governado pela Agentic AI Foundation (Linux Foundation); há um **spec RC 2026-07-28** com core stateless, framework de Extensions, Tasks virando extensão, e hardening de OAuth2/OIDC. Um SDD tool baseado em MCP server **tem de acompanhar isso ou quebra**.
- **AGENTS.md** (60k+ projetos), **APM** (Microsoft), **Claude Code plugin marketplace** e **Copilot custom agents** são as superfícies-padrão. O Specky já distribui via APM — um edge real e atual.

### Nuance de lineage (a corrigir — questão de credibilidade)

O README diz *"Powered by the [Spec-Kit](https://github.com/paulasilvatech/spec-kit) methodology"* e descreve Spec-Kit como tendo "EARS notation, gated pipeline phases, … 25+ specialized agents". Porém: (1) o link aponta para **`paulasilvatech/spec-kit`, um fork 0-star do `github/spec-kit`** sem divergência substancial visível; (2) o upstream **não usa EARS nem enforça fases** — isso vem da Kiro (EARS) e é contribuição própria do Specky (o motor enforçado). Rotear "a metodologia Spec-Kit" por um fork pessoal enquanto atribui a ele features que são da Kiro/do próprio Specky é o tipo de coisa que threads de HN punem — e é trivial de corrigir. **Posicionamento honesto:** *"O Specky estende o workflow do Spec-Kit (crédito `github/spec-kit`) e os requisitos EARS ao estilo Kiro, reimplementados como uma máquina de estados MCP enforçada."*

### Colisões de marca

Existem `specky` (lib de validação npm, 2016), **`specky.app`** (SaaS comercial "Specification-Driven Project Autopilot") e `speckit.org` — diluição de SEO a considerar no posicionamento.
