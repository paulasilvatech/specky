# Specky — Verified Test Loop — Specification

---

## 1. Test Execution (REQ-EXEC)

### REQ-EXEC-001: Test Runner Invocation via MCP (Event-driven)

When `sdd_verify_tests` is called and a compatible test runner MCP server is active, the system shall invoke the runner and parse structured JSON results.

**Acceptance Criteria:**
- Detects active MCP test runners via `sdd_check_ecosystem`
- Sends test execution request to runner MCP with spec_dir and framework
- Parses response into `TestResult[]` with `{ test_name, passed, error?, duration_ms }`
- Times out after 120 seconds with actionable error message

**Traces to:** SC-001 | **Evidence:** arXiv:2404.10100

---

### REQ-EXEC-002: CLI Fallback for Test Execution (Optional)

If no test runner MCP is detected, the system shall execute the test command via shell and parse stdout/stderr into structured results.

**Acceptance Criteria:**
- Detects framework from `sdd_scan_codebase` result
- Executes: `npx vitest run --reporter=json` or `python -m pytest --json-report` depending on framework
- Parses output into same `TestResult[]` format
- Documents which execution path was used in response

**Traces to:** SC-005, SC-006 | **Evidence:** arXiv:2404.10100

---

## 2. Results Mapping (REQ-MAP)

### REQ-MAP-001: Test-to-Requirement Traceability (Ubiquitous)

The system shall map each test result to its source requirement ID using the comment convention `// REQ-XXX-NNN` generated by `sdd_generate_tests`.

**Acceptance Criteria:**
- Reads generated test files and extracts `// REQ-` comments
- Maps each passing/failing test to its requirement ID
- Requirements with no mapped tests are flagged as `untested`
- One requirement can map to multiple tests

**Traces to:** SC-002 | **Evidence:** arXiv:2509.14745

---

### REQ-MAP-002: Acceptance Criteria Coverage Report (Ubiquitous)

The `sdd_verify_tests` response shall include per-requirement coverage showing which acceptance criteria have passing tests.

**Acceptance Criteria:**
- `coverage.per_requirement[req_id].status` is `passing`, `failing`, or `untested`
- `coverage.per_requirement[req_id].test_count` is the number of mapped tests
- `coverage.overall_percent` is requirements with passing tests / total requirements * 100
- `coverage.failing_requirements[]` lists IDs of requirements with failing tests

**Traces to:** SC-002, SC-003 | **Evidence:** arXiv:2404.10100

---

## 3. Failure Reporting (REQ-REPORT)

### REQ-REPORT-001: Failed Test Output in Response (Ubiquitous)

The `sdd_verify_tests` response shall include the first 500 characters of error output for each failing test.

**Acceptance Criteria:**
- `failures[].requirement_id` links to the traced requirement
- `failures[].test_name` is the test identifier
- `failures[].error_snippet` is max 500 chars of stderr/assertion output
- `failures[].suggested_fix_prompt` is a ready-to-use agent prompt for fixing the test

**Traces to:** SC-004 | **Evidence:** arXiv:2604.03196

---

## 4. Quality (REQ-QUALITY)

### REQ-QUALITY-001: Unit Tests for TestResultParser (Ubiquitous)

The test suite shall verify correct parsing of Vitest JSON, pytest JSON, and JUnit XML output formats.

**Acceptance Criteria:**
- 3 fixture files: one per format
- Parser produces identical `TestResult[]` from each fixture
- Edge cases: empty test suite, all passing, all failing

**Traces to:** SC-001, SC-005

---

### REQ-QUALITY-002: Traceability Mapping Tests (Ubiquitous)

The test suite shall verify requirement ID extraction from generated test files.

**Acceptance Criteria:**
- 3 tests: single REQ comment, multiple REQ comments, no REQ comment
- Untested requirements correctly flagged
- Overall coverage percent calculated correctly

**Traces to:** SC-002
