# Specky — Constitution

> The foundational charter for the Specky project, establishing principles, constraints, and success criteria.

---

## Article 1: Project Identity

- **Name:** Specky
- **Description:** Spec-Driven Development CLI toolkit with 58 MCP tools, 13 agents, 22 prompts, 14 skills, 16 hooks
- **Creator:** Paula Silva
- **License:** MIT

---

## Article 2: Principles

- Every public API change must be accompanied by tests
- All user-facing numbers must be generated by script, never hardcoded
- The tool must practice what it preaches: use SDD for its own development
- Minimal permissions by default; explicit opt-in for elevated access
- Deterministic output: same inputs produce same artifacts

---

## Article 3: Constraints

- No runtime dependencies beyond @modelcontextprotocol/sdk, yaml, zod
- No outbound network calls from the MCP server
- All file I/O must go through FileManager for path sanitization
- State mutations must be atomic and signed

---

## Article 4: Success Criteria

| ID | Criterion | Measure |
|----|-----------|---------|
| SC-001 | Project compiles without errors | `npm run build` exits 0 |
| SC-002 | All tests pass | `npm test` exits 0 |
| SC-003 | Coverage thresholds met | `npm run test:coverage` exits 0 |
| SC-004 | Lint clean | `npm run lint` exits 0 with 0 warnings |
| SC-005 | All requirements traceable | Every REQ has design + task mapping |

---

## Article 5: Scope

### In Scope
- Biome lint integration for code quality
- Test coverage improvements for uncovered modules
- Dogfooding: spec package for Specky itself

### Out of Scope
- New MCP tools or agents
- Breaking API changes
- New runtime dependencies
