# Specky Examples

Real, complete spec packages produced by running the Specky pipeline
end-to-end — not hand-written mock-ups. Each was generated by driving the MCP
server through `sdd_init → sdd_discover → sdd_write_spec → sdd_write_design →
sdd_write_tasks → sdd_run_analysis`, exactly as an AI IDE would.

## `todo-api/` — greenfield REST API

A small but complete feature package for a to-do REST API, under
[`todo-api/001-todo-api/`](todo-api/001-todo-api/):

| Artifact | What it shows |
|----------|---------------|
| `CONSTITUTION.md` | Project principles and constraints |
| `SPECIFICATION.md` | 4 requirements in EARS notation (ubiquitous, event-driven, unwanted) with acceptance criteria |
| `DESIGN.md` | Architecture, data model, security notes, Mermaid diagrams, and requirement → component traceability |
| `TASKS.md` | Six tasks with dependencies and `traces_to` back to requirements |
| `ANALYSIS.md` | The **real quality gate**: `APPROVE` at 100% coverage, 0 orphaned requirements, 100% EARS compliance |
| `DIAGRAMS.md`, `ADR.md`, `PLAYBOOK.md`, `TDD_STATUS.md`, `EVIDENCE.md`, `SPEC_PACKAGE.json` | Companion artifacts generated with the spec |

### Why the gate is meaningful here

`ANALYSIS.md` reports `APPROVE` **because** every requirement is traced through
both design and tasks. The first generation of this package returned
`CHANGES_NEEDED` at 75% coverage — the `DESIGN.md` did not reference the
requirement IDs, so the traceability check flagged them as unmapped. Adding the
requirement → component mapping to the design is what moved the gate to
`APPROVE`. In other words, the gate is computed from the artifacts, not asserted
— the same `AnalysisEngine` runs for `sdd_run_analysis` and the auto-pipeline.

## Reproduce it

From any empty directory, drive the same tools through your AI IDE (or a raw
MCP client) in order. The pipeline blocks phase-skipping and refuses to advance
past a scaffolded `DESIGN.md`/`TASKS.md` until real content replaces it, so the
package you get is always internally consistent.
